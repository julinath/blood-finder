01

The Problem

Finding blood fast is too hard

⏱️
No reliable place to search

A saved phone number may not answer when it matters; Facebook groups are slow and posts sink; local blood clubs don't share their lists.

💸
Some make money from the crisis

People sell donor phone numbers, fake donors ask for money at the hospital, and some even resell donated blood.

🔓
No privacy for donors

Phone numbers get spread across public groups — fear of spam and unwanted calls keeps willing donors away.

😨
Fear and wrong beliefs

"Donating makes you weak" — myths and missing knowledge stop thousands of healthy people from ever donating.

“Someone in Bangladesh needs blood every two seconds — yet there is no reliable place to look.”

1 bag of blood can save up to 3 lives
Blood cannot be manufactured — only donated

The Solution

One platform, three flows

FLOW — 01

🔍

Find Donors

Search verified donors by blood group, district and area in seconds — with a 90-day eligibility badge and donation count, so you instantly know who can give.

FLOW — 02

🚑

Emergency Board

A patient's family posts a request; nearby donors step forward with one tap — “আমি রক্ত দিতে পারবো”. The feed auto-filters to the viewer's own blood group and district.

FLOW — 03

🩸

Become a Donor

One form to register as a donor — visible in search only after admin approval. Every donation is recorded on the platform and counts toward the donor's public profile.

The Product

Made for everyday phone users

04

Find Donors

From search to contact in seconds

✓ Verified-only results

Only admin-approved donors ever appear in search — fake profiles have no place here.

✓ 90-day eligibility rule

Counted from the last donation date and applied everywhere — search badges, profiles, and the request form.

✓ Contact gated by login

Phone numbers require sign-in. Anonymous visitors cannot read them even directly from the database.

05

Emergency Board · our signature design

Donor-first privacy

1 · Request posted

The contact number is stored in a separate, access-controlled table — it never appears on the public feed.

2 · A donor responds

One tap — “আমি রক্ত দিতে পারবো” — but the donor never sees the requester's number.

3 · The requester calls

The requester sees the responding donor's name and number, and calls them. Donor privacy comes first — no spam, no harassment.

06

Request Lifecycle

From request to a recorded donation

PENDING ──▶donor accepts ACCEPTED ──▶requester confirms “রক্ত পেয়েছি” COMPLETED ⤷ CANCELLED
Accept = consent

Acceptance reveals the donor's number to the requester — they talk and agree on time and place.

The receiver confirms — not the donor

Only the requester can mark it complete. A donor can never confirm their own donation — so nobody can inflate their own count. Same rule as the emergency board.

Numbers that can't lie

One database function closes the request and saves the donation record together; the database updates the count and the 90-day countdown on its own.

07

Trust & Safety

The admin panel — keeps it safe & real

✓ Donor approval queue

Each application arrives with an automatic fitness verdict (age, weight, donation-gap rules); admins approve, reject, or un-list donors.

✓ Reports queue

No-shows, payment demands, fake requests — user reports flow through an Open → Reviewed → Resolved pipeline.

✓ Full oversight

Expire stale emergencies, manage users and requests — every admin action is double-guarded at both server and database level.

08

Tech Stack

What we use — and why

Next.js 16
+ React 19 Web framework — pages, routing & server code in one place UI and server logic in one framework — no separate backend to build and maintain.
TypeScript Typed JavaScript — our whole domain model is typed Mistakes are caught while writing code, not after deploy. Donor, BloodRequest — every model has a type.
Tailwind CSS 4 Styling system One consistent design language, tiny CSS output, very fast to build mobile-first UI.
Supabase PostgreSQL database + Auth (login) Real SQL with Row Level Security — data protected inside the database itself. Email, mobile-number & Google login readymade.
Vercel Hosting + CI/CD Every git push auto-deploys. Free, fast, global — perfect for a student project that wants to go real.
+ tools d3-geo (district map) · Playwright (browser testing) · Hind Siliguri (Bangla font) The map is drawn on the server; every flow is tested in a real browser.

Defense in depth

UI validation → server-action validation → database constraints & RLS. Even if one layer fails, the data stays safe.

Performance

The 700 KB district map is projected to SVG on the server — it never reaches the phone. CSS-only animations, reduced-motion respected.

Quality process

Every flow tested end-to-end in a real browser with Playwright (desktop + 390px mobile); lint, type-check and production build all clean.

OOP in Blood Finder · the course connection

Same OOP ideas, two stacks

EncapsulationKeep data private; change it only in safe ways

Desktop

Donor.java hides its fields — no other class can change them directly.

Web

One file (eligibility.ts) holds the 90-day rule; the whole site uses that one place.

AbstractionHide the messy details behind a simple call

Desktop

A Repository hides the SQL — the rest of the app just asks for data.

Web

Server Actions hide the database — the UI just calls acceptRequest().

Inheritance & PolymorphismReuse code — one shape, many forms

Desktop

A shared base class; swap the part underneath and the rest of the code stays the same.

Web

One BloodTypeBadge / Field component, reused in many places.

MVC — Separation of ConcernsSplit the app into clear, separate parts

Desktop

Model (data) · View (FXML screen) · Controller (logic).

Web

Database = Model · React pages = View · Server actions = Controller.
